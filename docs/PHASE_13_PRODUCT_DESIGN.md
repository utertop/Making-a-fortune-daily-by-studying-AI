# Phase 13 产品设计稿：云端化、桌面化、归档视图与推送闭环

本文档用于审核下一阶段设计。目标是先把业务逻辑讲通，再进入实现。

当前结论：v0.1 已经能本地跑通，但它仍然依赖人工启动、本地单机状态和半自动文档提交。下一阶段不应该只继续堆按钮，而应该把系统升级成“持续运行的 AI 学习工作台”。

## 1. 阶段目标

Phase 13 需要解决四类问题：

1. 云端部署和 PC / Web 使用形态。
2. 左侧时间归档导航，支持按日 / 周查看历史信号和处理状态。
3. 推送规则产品化，包括定时推送、状态变更通知、deadline 提醒。
4. 重新设计按钮背后的业务状态机，让“是否完成今日任务”有清晰规则。

## 2. 部署形态设计

### 2.1 当前问题

现在必须人工执行：

```powershell
scripts\start_local.cmd
```

这适合开发调试，不适合作为长期使用产品。

### 2.2 目标形态

系统最终应支持三种使用方式：

| 形态 | 用途 | 推荐阶段 |
| --- | --- | --- |
| 本地开发版 | 开发、调试、离线使用 | 已有，继续保留 |
| 云端 Web 版 | 随时访问、定时任务、推送稳定运行 | Phase 13 重点 |
| PC 桌面版 | 开机可用、托盘常驻、本地文件集成 | Phase 14 或 Phase 13 后半 |

### 2.3 云端部署建议

推荐组合：

| 模块 | 推荐方案 | 原因 |
| --- | --- | --- |
| Web 页面 | Vercel / Cloudflare Pages | 前端部署简单，访问稳定 |
| API 服务 | Render / Railway / Fly.io / 自有服务器 | 需要跑后端和定时任务 |
| 数据库 | Supabase Postgres | 免费额度可用，后续团队化方便 |
| 定时任务 | GitHub Actions / Supabase Edge Functions / 云服务器 cron | 支持 8:00、14:00、21:30 定时 |
| 文件存储 | GitHub repo / Supabase Storage / S3 兼容存储 | 保存 Markdown、附件、图片 |

第一版云端不建议一上来做太复杂。推荐先做：

```text
Vercel Web + Supabase DB + GitHub Actions 定时采集/推送
```

原因：改动较小，成本低，容易回滚。

### 2.3.1 成本与免费层判断

对个人 MVP 来说，前端托管通常可以先免费起步，但完整的 AI Signal Radar 不太可能永远零成本。

#### Vercel

Vercel 适合放前端 Web 页面，优势是：

1. 部署 Next.js 简单。
2. GitHub 集成顺滑。
3. 访问体验稳定。

当前判断：

1. 个人项目可以先用免费层起步。
2. 适合先部署 Web 页面和轻量前端逻辑。
3. 如果后续流量、函数调用、构建时间、团队协作需求上来，大概率需要升级付费。

产品判断：

```text
前端页面先上 Vercel 免费层是可行的。
```

#### Cloudflare Pages / Workers

Cloudflare 更适合：

1. 静态页面托管。
2. 轻量 Functions / Workers。
3. 低成本边缘请求处理。

当前判断：

1. 静态页面可以先免费。
2. 如果用了 Pages Functions / Workers，也有免费额度。
3. 一旦请求量、函数调用、数据库访问和调度增多，后续会进入付费范围。

产品判断：

```text
Cloudflare 也能先免费起步，但它更像“边缘基础设施方案”，不是只给前端页面用。
```

#### 对本项目的现实建议

需要拆开看：

| 模块 | 是否能先免费 | 说明 |
| --- | --- | --- |
| 前端 Web 页面 | 可以 | Vercel / Cloudflare Pages 都可先免费起步 |
| 后端 API | 视平台而定 | 往往会遇到运行时、休眠、配额限制 |
| 定时任务 | 可以先低成本 / 免费尝试 | GitHub Actions 可先承担部分调度 |
| 数据库 | 可以先免费额度 | Supabase 免费层足够做 MVP，但后期可能受限 |
| 飞书推送 | 可以 | 主要不是平台费，而是后端执行环境成本 |

结论：

```text
“页面上线”可以先不付费；
“完整系统上线”大概率后续会有成本。
```

因此推荐策略是：

1. 第一阶段先把前端页面和基础 API 云端跑起来。
2. 定时任务先用低成本方案承担。
3. 等业务规则稳定后，再评估是否升级付费套餐。

不建议一开始就把“零成本永久运行”作为前提，因为：

1. 定时推送需要稳定执行环境。
2. 飞书通知、状态同步、数据库读写、周汇总都会消耗资源。
3. 以后接入 API key 自动总结后，成本一定会上升。

#### 当前文档结论

对于 Phase 13：

```text
可以按“前端先免费部署、后端和调度按 MVP 最低成本配置”的思路推进。
```

### 2.4 PC 桌面版建议

桌面版有两条路线：

| 路线 | 说明 | 建议 |
| --- | --- | --- |
| Tauri | 轻量、性能好、安装包小 | 推荐 |
| Electron | 生态成熟，但体积大 | 可选 |

PC 版应该解决：

1. 一键安装。
2. 开机自启动。
3. 托盘常驻。
4. 本地 Markdown 文件夹选择。
5. 本地和云端同步状态展示。

但桌面版不应先于云端状态模型。建议先完成云端数据模型和业务状态机，再做桌面壳，否则容易返工。

### 2.5 本地与云端同步

后续需要从“纯本地 SQLite”升级成“双存储模型”：

```text
Local SQLite / local files
        ↕ sync
Cloud Postgres / cloud object storage
```

推荐同步原则：

1. 云端是任务状态和推送状态的主记录。
2. 本地是 Markdown 编辑体验和离线缓存。
3. 每条任务、文档、推送事件都必须有稳定 ID。
4. 同步冲突先不自动合并，先显示冲突，由用户选择保留本地或云端。

## 3. 页面左侧知识库归档导航

### 3.1 当前问题

当前页面只展示今日任务，没有长期知识库视角。

下一阶段左侧导航不应该只是简单的日期列表，而应该是一个“AI 知识库目录树”。它的目标是让用户像翻笔记一样查看历史，同时保留任务处理状态。

### 3.2 目标目录结构

左侧目录按真实日历动态生成，结构示例：

```text
2026年AI知识库
  5月AI知识库
    第一周
      5月1日知识库
      5月2日知识库
      5月3日知识库
      5月4日知识库
      5月5日知识库
      本周汇总
    第二周
      5月6日知识库
      5月7日知识库
      ...
      本周汇总
    ...
    第六周
      5月31日知识库
      本周汇总
```

具体年份、月份、每个月包含几周，不写死，由日历逻辑自动计算。

### 3.3 日历生成规则

1. 年份来自系统部署后的实际数据，不展示部署前没有数据的年份。
2. 月份来自已有采集 / 推送 / 文档记录，不展示完全没有数据的月份。
3. 每个月的周数由日历计算，可能是 4、5 或 6 周。
4. 每周只展示该月内实际存在数据的日期。
5. 如果某一天没有任何采集、推送或文档记录，则默认不展示。
6. 历史从系统开始部署和采集的日期算起，没有部署就没有历史记录，也不应该伪造知识库。

建议周划分采用“月内周”而不是 ISO 周：

```text
第一周 = 当月 1 日所在周在本月内覆盖的日期
第二周 = 下一段自然周
...
```

这样用户看到的目录更符合“5月第一周、5月第二周”的直觉。

### 3.4 页面布局建议

主内容不变，新增左侧知识库目录：

```text
┌────────────────────────────┬──────────────────────────────────────┐
│ 左侧知识库目录              │ 主内容区                               │
│                            │                                      │
│ 2026年AI知识库              │ 指定日 / 指定周的 AI 信号和知识库状态    │
│   5月AI知识库               │ 统计卡片                               │
│     第一周                  │ 任务卡片                               │
│       5月1日知识库           │ 文档状态                               │
│       5月2日知识库           │ 操作按钮                               │
│       本周汇总              │                                      │
└────────────────────────────┴──────────────────────────────────────┘
```

### 3.5 日知识库视图

点击“5月1日知识库”后，主内容展示这一天的完整记录：

1. 当日采集的 AI 信号总数。
2. 当日推送数量。
3. 当日选择跟进数量。
4. 当日已提交知识文档数量。
5. 当日忽略数量。
6. 当日未决策数量。
7. 当日未完成跟进数量。
8. 每条信号的完整状态。

每条信号至少展示：

1. 标题和来源链接。
2. 评分和触发原因。
3. 推送状态。
4. 是否选择跟进。
5. 是否创建草稿。
6. 是否提交知识文档。
7. 是否忽略 / 归档。
8. 文档路径。

### 3.6 本周汇总视图

点击“本周汇总”后，主内容展示本周内所有列出的项目。

排序规则：

```text
按 signal_score 从高到低排序
```

如果分数相同，建议依次按：

1. 是否已选择跟进。
2. 是否已提交文档。
3. GitHub star 增长。
4. 最近采集时间。

本周汇总展示内容：

1. 本周 Top Signals。
2. 本周已文档化项目。
3. 本周未完成跟进项目。
4. 本周忽略项目及原因。
5. 本周高频关键词。
6. 本周值得持续跟进的项目。
7. 本周学习完成率。

半自动阶段，本周汇总可以先是规则聚合页面；全自动阶段再由 LLM 生成自然语言周报。

### 3.7 数据要求

为了支持日 / 周 / 月 / 年归档，需要为数据补充字段：

| 字段 | 用途 |
| --- | --- |
| collected_at | 采集时间 |
| pushed_at | 推送时间 |
| selected_at | 选择跟进时间 |
| documented_at | 提交文档时间 |
| ignored_at | 忽略时间 |
| archived_at | 归档时间 |
| work_date | 归属日期，按 Asia/Shanghai |
| work_week | 归属周 |
| work_month | 归属月份 |
| work_year | 归属年份 |
| month_week_index | 月内第几周 |

`work_date`、`work_month`、`work_year`、`month_week_index` 可以在查询时动态计算，也可以在写入任务时冗余保存。云端化后建议冗余保存，方便查询和同步。

## 4. 推送业务设计

### 4.1 推送类型

建议把推送拆成四类：

| 类型 | 触发方式 | 内容 |
| --- | --- | --- |
| 定时推荐推送 | 8:00、14:00 | Top AI Signals |
| 操作状态推送 | Web 按钮点击 | 跟进、文档提交、忽略、归档等变化 |
| soft deadline 提醒 | 21:30 | 今日完成情况和未完成项 |
| hard deadline 提醒 | 23:00 | 今日最终完成情况和未完成项 |
| 昨日未完成追踪 | 次日首次任务推送 | 昨日 hard deadline 后仍未完成项，仅提醒一次 |
| 周总结推送 | 每周固定时间 | 本周学习总结和沉淀文档 |

### 4.2 定时推荐规则

默认规则：

| 时间 | 工作日 | 周末 |
| --- | --- | --- |
| 8:00 | 推送 Top 10 | 推送 Top 5 |
| 14:00 | 新增 / 变化 Top 5，或进度提醒 | 新增 / 变化 Top 3，或进度提醒 |
| 21:30 | soft deadline 提醒 | soft deadline 提醒 |
| 23:00 | hard deadline 提醒 | hard deadline 提醒 |

注意：14:00 不应该简单重复 8:00 内容。

每日新增推送上限：

| 类型 | 每日最多新推 |
| --- | --- |
| 工作日 | 15 条 |
| 周末 | 8 条 |

这里的“新推”指当天首次出现在飞书推荐里的信号，不包括 deadline、状态变更通知和昨日未完成回顾。

设计原因：

1. 8:00 是主推批次，负责覆盖当天最重要的 AI 信号。
2. 14:00 是补充批次，负责新增高分信号、热度继续上升的信号和上午遗留状态。
3. 如果下午没有足够新增内容，不硬凑数量，只发送进度提醒。
4. 周末默认降噪，避免学习系统变成负担。

14:00 推送优先级：

1. 8:00 后新增的高分信号。
2. 上午已推送但热度继续上升的信号。
3. 上午被用户选择跟进但尚未提交文档的信号。
4. 如果没有新内容，则发送简短状态提醒，而不是硬凑 10 条。

### 4.3 飞书推送内容

推荐推送应包含：

1. 今日批次：上午 / 下午。
2. 每条信号标题。
3. 分数和触发原因。
4. 风险提示。
5. 源链接。
6. Web 工作台入口。
7. 今日完成进度。

### 4.4 状态变更推送

页面按钮点击后，不建议所有状态都即时推送飞书，否则容易刷屏。

推荐采用“推送分级”：

| 操作 | 飞书通知 |
| --- | --- |
| 选择跟进 | 立即推送 |
| 提交文档 | 立即推送 |
| 生成 Markdown 草稿 | 默认不推送，只做页面提示 |
| 忽略 | 默认不推送，可汇总到后续状态摘要 |
| 归档 | 默认不推送，可汇总到后续状态摘要 |

#### 4.4.1 高优先级即时推送

建议只有两类操作即时推送：

1. 选择跟进。
2. 提交知识文档。

原因：

1. 这两类操作真正改变了今日任务完成路径。
2. 它们对飞书提醒最有价值。
3. 它们能够帮助用户确认关键进展。

#### 4.4.2 低优先级状态不即时推送

以下操作默认不发飞书即时通知：

1. 创建知识库草稿。
2. 忽略。
3. 归档。

这些操作可以：

1. 只在 Web 页面提示成功。
2. 或者被汇总到 14:00、21:30 的状态摘要里。

#### 4.4.3 后端幂等规则

重点不在“5 分钟内去重”，而在于后端幂等。

规则：

1. 如果任务已经是目标状态，再次点击不应产生新事件。
2. 不写新的 task_event。
3. 不发新的飞书通知。
4. 前端按钮应置灰、隐藏或提示“状态未变化”。

这样可以从根本上避免重复点击造成的刷屏。

#### 4.4.4 连续关键操作合并推送

如果用户在很短时间内连续完成多个关键步骤，例如：

```text
选择跟进 -> 创建知识库草稿 -> 提交知识文档
```

可以合并为一条关键更新，而不是三条飞书消息。

示例：

```text
项目 xxx 状态更新：
已选择跟进 -> 已创建知识库草稿 -> 已提交知识文档
```

默认策略建议：

1. 如果 60 秒内连续完成关键动作，优先合并推送。
2. 如果只有单个关键动作，则即时推送单条消息。

#### 4.4.5 与 deadline 的关系

deadline 只负责提醒未完成项，不重复提醒已完成任务。

规则：

1. 已提交文档的任务，不进入 deadline 未完成提醒。
2. 已忽略 / 已归档的任务，不进入 deadline 未完成提醒。
3. 只有 `selected` 未提交文档，或 `pending_decision` 未决策的任务，进入 deadline 提醒。

### 4.5 deadline 提醒

deadline 分为 soft deadline 和 hard deadline。

| 时间 | 类型 | 规则 |
| --- | --- | --- |
| 21:30 | soft_deadline | 第一次正式提醒今日任务完成情况 |
| 23:00 | hard_deadline | 最后一次提醒今日任务完成情况 |
| 次日首次推送 | yesterday_carryover | 只在新一天第一次任务推送时，顺带提醒昨日 hard deadline 后仍未完成项 |

#### 4.5.1 soft deadline

每日 21:30 发送 soft deadline。

内容必须详细说明：

1. 今日哪些任务已经完成。
2. 今日哪些任务仍未完成。
3. 哪些项目已选择跟进但未提交文档。
4. 哪些项目还处于未决策状态。
5. Web 工作台入口。

示例：

```text
今日 AI 学习进展提醒（soft deadline）

候选：10
已选择跟进：3
已提交文档：2
已完成跟进：2
未完成跟进：1
未决策：2
已忽略：5
已归档：2

已完成项：
1. xxx：已提交知识文档
2. yyy：已提交知识文档

未完成项：
1. aaa：已选择跟进，但未提交文档
2. zzz：尚未决策是否跟进

请在 23:00 hard deadline 前完成处理。
```

#### 4.5.2 hard deadline

每日 23:00 发送 hard deadline。

这是当天最后一次 deadline 通知，不再继续提醒。

内容必须详细说明：

1. 当天最终完成情况。
2. 已完成项。
3. 未完成项。
4. 未完成原因分类：未决策 / 已跟进未提交文档。
5. 明确说明这是当天最后一次提醒。

示例：

```text
今日 AI 学习最终提醒（hard deadline）

候选：10
已完成文档：2
未完成跟进：1
未决策：2
已忽略：5

已完成项：
1. xxx：已提交知识文档
2. yyy：已提交知识文档

仍未完成：
1. aaa：已选择跟进，但未提交文档
2. bbb：尚未决策是否跟进
3. ccc：尚未决策是否跟进

这是今天最后一次 deadline 提醒。
如果未完成，明天第一次任务推送时会附带昨日未完成摘要。
```

#### 4.5.3 次日首次推送携带昨日未完成

如果某天 hard deadline 后仍有未完成项，次日第一次任务推送时需要一并提醒。

规则：

1. 只在新一天第一次任务推送时提醒。
2. 后续 14:00 推送、状态变更推送、当天 deadline 不再重复提醒昨日项。
3. 如果昨日未完成项已经在次日首次推送前被补完，则不再提醒。
4. 这类提醒应作为新一天推送的一个独立区块，不要覆盖当天任务内容。

示例：

```text
昨日未完成回顾

昨天 hard deadline 后仍未完成：
1. aaa：已选择跟进，但未提交文档
2. bbb：尚未决策是否跟进

本提醒仅在今天第一次任务推送时出现一次。
```

deadline 的核心检查不应该是“所有候选都提交文档”，而应该是：

```text
所有 selected 的任务是否都有 knowledge document。
```

同时还需要检查：

```text
所有 pending_decision 的任务是否已经完成决策。
```

## 5. 按钮业务逻辑重设计

### 5.1 核心原则

今日任务完成不应该要求所有推送信号都写文档。否则会变成负担。

建议规则：

```text
只有用户选择跟进的任务，才进入必须提交文档的完成检查。
```

候选信号有三种合理结局：

1. 选择跟进，之后提交文档。
2. 忽略，不进入今日未完成。
3. 保持未处理，deadline 时提醒“仍未决策”。

### 5.2 状态机

建议任务状态：

| 状态 | 含义 | 是否算未完成 |
| --- | --- | --- |
| discovered | 被系统发现，还未进入今日任务 | 否 |
| pushed | 已被定时推送 | 否 |
| pending_decision | 等待用户判断是否跟进 | 是，属于未决策 |
| selected | 用户选择跟进 | 是，必须提交文档 |
| draft_created | 已生成 Markdown 草稿 | 是，仍需提交正式文档 |
| review_pending | 已检测到知识库文档更新，等待人工确认提交 | 是，仍需确认 |
| documented | 已提交知识库文档 | 否 |
| ignored | 明确不处理 | 否 |
| archived | 已归档 | 否 |

v0.1 当前的 `pending` 可以逐步替换成 `pending_decision`，语义更清楚。

### 5.3 按钮设计

#### 标记已推送

不建议继续作为人工按钮。

新设计：

1. 定时推送成功后，系统自动把任务状态变为 `pushed` 或记录 `pushed_at`。
2. 页面展示“已推送”标签。
3. 按钮置灰或直接不显示。

人工不应该负责“标记已推送”，否则会和真实推送状态冲突。

#### 选择跟进

这是核心人工决策按钮。

点击后：

1. 状态变为 `selected`。
2. 进入今日完成检查。
3. 飞书发送“已选择跟进”通知。
4. 页面提示下一步：生成草稿或提交文档。

业务规则：

```text
selected 后，如果到 21:30 仍无文档，则提醒未完成。
```

#### 忽略

保留“忽略”，并增加原因：

1. 与方向不相关。
2. 质量不足。
3. 已了解。
4. 暂时不重要。
5. 重复信号。

点击后：

1. 状态变为 `ignored`。
2. 不进入今日未完成。
3. 记录原因，用于操作留痕、周报统计和后续人工复盘。
4. 默认不直接进入自动评分逻辑。
5. 默认不即时推送飞书，可汇总到后续状态摘要。

#### 生成 Markdown 草稿

这个按钮仍然有必要，但定位要调整。

半自动阶段：

1. 它生成标准模板，减少每次新建文档的摩擦。
2. 它提供固定路径，方便 Codex / Antigravity 直接补全。
3. 它让系统知道这个任务已经进入“文档处理中”。

全自动阶段：

1. 它会变成 Agent 生成文档的前置结构。
2. LLM 可以基于模板稳定输出。
3. 生成的 Markdown 更容易保持统一格式。

结论：

```text
保留，但改名为“创建知识库草稿”更准确。
```

推荐流程：

```text
选择跟进 -> 创建知识库草稿 -> 用 Codex/Antigravity 或 API Agent 补全并写回目录 -> review_pending -> 确认提交文档
```

#### 文档检测与确认提交

半自动阶段里，“文件已经写进目录”不等于“已经正式提交文档”。

需要拆成两个阶段：

1. 文档写回目录。
2. 人工确认提交。

业务原因：

1. 补全文档可能还没有审核。
2. 目录里可能只是初稿或半成品。
3. 如果系统只因为文件被改动就自动算完成，会让任务状态失真。

建议新增状态：

```text
review_pending
```

含义：

```text
系统已检测到目标 Markdown 文件有实质更新，但仍等待人工确认提交。
```

页面需要明确区分两类提示：

1. 已生成草稿，但暂未检测到补全文档。
2. 已检测到待确认知识库文档。

页面提示示例：

```text
已生成草稿，暂未检测到补全文档
```

```text
检测到待确认知识库文档
路径：knowledge-base/daily/...
最后更新时间：15:42
```

#### 检测规则

不能只靠“文件存在”判断，因为草稿生成时文件就已经存在。

建议同时检查：

1. 文件修改时间是否晚于 `draft_created_at`
2. 文件内容 hash 是否与初始草稿 hash 不同
3. 是否存在实质内容更新，而不是轻微格式变化

建议记录：

```text
draft_created_at
draft_initial_hash
last_detected_hash
last_detected_at
```

满足以下条件时，状态可变为 `review_pending`：

```text
文件存在
且修改时间晚于 draft_created_at
且 hash 已变化
```

#### 检测频率与时机

不要高频死轮询，使用“关键时机检查 + 轻量轮询”。

立即检查时机：

1. 页面打开时
2. 切换到某一天 / 某一周时
3. 点击“创建知识库草稿”后
4. 页面重新获得焦点时
5. 用户点击“手动刷新状态”时

后台轮询规则：

1. 仅在页面打开且可见时轮询
2. 默认每 90 秒检查一次
3. 页面隐藏时停止轮询
4. 页面重新可见时立即补查一次

选择 90 秒的原因：

1. 对半自动流程足够及时
2. 不会给本地文件和后端带来明显压力
3. 不会因为高频刷新让页面显得焦躁

#### 确认提交文档

这是今日完成的关键按钮。

点击后：

1. 验证文档路径存在。
2. 写入 knowledge_document。
3. 状态从 `review_pending` 变为 `documented`。
4. 飞书推送“已提交文档”。
5. 从今日未完成列表移除。

如果文档不存在，不能算提交成功。

页面按钮文案建议从“提交文档”改为：

```text
确认提交文档
```

因为页面此时不是“上传文件”，而是对已写回目录的知识库文档做人工确认。

#### 归档

归档不应该代替文档提交。

建议规则：

1. `ignored` 的任务不需要归档。
2. `documented` 的任务可以归档。
3. `selected` 但未提交文档的任务不允许直接归档，除非选择“放弃跟进”。
4. `pushed / pending_decision` 可以归档为“仅归档未跟进”，但需要明确不计入学习完成。

更清晰的按钮设计：

| 当前按钮 | 建议按钮 |
| --- | --- |
| 忽略 | 忽略 |
| 归档 | 归档已完成 |
| 生成 Markdown 草稿 | 创建知识库草稿 |
| 提交文档 | 提交知识文档 |

### 5.4 今日完成判断

建议公式：

```text
今日完成 = selected_count > 0 且 selected_without_document_count = 0
```

但如果当天没有选择任何跟进，也不能直接算“学习完成”。应显示：

```text
今日无跟进项，请确认是否至少选择 1 个值得学习的信号，或全部标记忽略。
```

更完整的规则：

1. 所有候选必须完成决策：选择跟进 / 忽略 / 归档。
2. 所有选择跟进的任务必须提交文档。
3. 至少有 1 个 documented，才算有效学习日。

如果某天确实没有值得跟进的项目，可以允许用户点击：

```text
今日无值得跟进项
```

这会生成一条日总结记录，避免系统永远提醒。

## 6. 草稿文档是否必要

你的问题是对的：半自动下，是先生成草稿再喂给 Codex / Antigravity 好，还是直接让 Codex / Antigravity 生成完整知识文档好？

建议答案：

```text
草稿仍然必要，但不应该只是“简单说明文档”，而应该是“任务上下文 + 标准模板 + 输出约束”。
```

### 6.1 半自动阶段

草稿价值：

1. 固定文件路径。
2. 固定文档结构。
3. 保留系统评分、风险、触发原因。
4. 让 Codex / Antigravity 明确知道要补什么。
5. 让 Web 工作台知道该任务已经进入文档处理。

推荐使用方式：

```text
系统创建草稿 -> 用户让 Codex / Antigravity 基于草稿和链接补全
```

### 6.2 全自动阶段

草稿会变成 Agent 的上下文约束：

1. 统一文档格式。
2. 控制 token 输出结构。
3. 降低生成结果跑偏概率。
4. 便于后续 Mermaid、截图、引用链接自动补充。

所以不建议删除草稿功能，而是升级它。

## 8. 半自动与全自动路线

### 8.1 半自动阶段

保留人工决策：

1. 系统推送信号。
2. 用户选择跟进 / 忽略。
3. 系统创建草稿。
4. 用户用 Codex / Antigravity 补全文档。
5. 用户提交文档。
6. 系统做 deadline 提醒和周总结。

### 8.2 全自动阶段

接入个人 API key 后：

1. 系统自动阅读链接 / repo / docs。
2. 自动生成知识文档。
3. 用户审核和修订。
4. 系统推送“待审核文档”。
5. 用户确认后归档。

全自动阶段新增状态：

| 状态 | 含义 |
| --- | --- |
| agent_running | Agent 正在阅读和生成 |
| draft_ready | Agent 文档草稿已生成 |
| review_required | 等待用户审核 |
| approved | 用户确认 |

## 12. 当前建议结论

下一阶段不要直接做云端部署，也不要直接做桌面壳。

建议先做：

```text
Phase 13.1：业务状态机和按钮逻辑改造
```

原因：

1. 云端和桌面都依赖稳定状态机。
2. 推送也依赖 task_event 和 push_event。
3. 左侧归档也依赖清晰的日期和周维度。

```text

```

### 13.3 状态流实现

#### 13.3.1 主状态流

```text
pending_decision
  -> selected
  -> draft_created
  -> review_pending
  -> documented
  -> archived
```

或：

```text
pending_decision
  -> ignored
```

#### 13.3.2 关键规则

1. `selected` 后才允许创建草稿。
2. `draft_created` 不代表文档完成。
3. 检测到 Markdown 文件有实质更新后，状态进入 `review_pending`。
4. 只有人工点击“确认提交文档”，状态才进入 `documented`。
5. `documented` 后才允许“归档已完成”。

### 13.4 本地文档检测逻辑

#### 13.4.1 检测输入

检测器至少需要：

```text
target_doc_path
draft_created_at
draft_initial_hash
```

#### 13.4.2 检测规则

当满足以下条件时，认为文档已进入待确认状态：

1. 目标文件存在。
2. 文件修改时间晚于 `draft_created_at`。
3. 当前文件 hash 不等于 `draft_initial_hash`。

满足后：

```text
status = review_pending
detection_status = updated
review_pending_at = now
```

否则：

```text
status 保持 draft_created
detection_status = unchanged
```

#### 13.4.3 检测 API

建议新增：

```text
POST /tasks/{task_id}/detect-document
POST /tasks/detect-documents
```

用途：

- 单任务检测
- 当前视图任务批量检测

响应至少返回：

```json
{
  "task_id": 12,
  "status": "review_pending",
  "detection_status": "updated",
  "document_path": "knowledge-base/daily/2026/04/2026-04-23/example.md",
  "last_modified_at": "2026-04-23T15:42:00+08:00"
}
```

### 13.5 前端页面改动

#### 13.5.1 Today Workspace 新提示

页面需要增加两类提醒：

1. 草稿已创建但未检测到补全文档：

```text
已生成草稿，暂未检测到补全文档
```

2. 检测到待确认知识库文档：

```text
检测到待确认知识库文档
路径：knowledge-base/daily/...
最后更新时间：15:42
```

#### 13.5.2 顶部状态汇总

建议增加：

1. 草稿处理中数量
2. 待确认文档数量
3. 已确认文档数量

#### 13.5.3 按钮调整

| 当前按钮 | Phase 13.1 按钮 |
| --- | --- |
| 生成 Markdown 草稿 | 创建知识库草稿 |
| 提交文档 | 确认提交文档 |

并新增：

```text
刷新检测状态
```

用于手动触发一次文件检查。

### 13.6 检测频率

#### 13.6.1 自动检测时机

页面应在以下时机触发检测：

1. 页面首次加载
2. 切换日期 / 周视图
3. 创建知识库草稿后
4. 页面重新获得焦点时
5. 用户手动点击刷新

#### 13.6.2 轮询策略

仅当页面可见时：

```text
每 90 秒轮询一次
```

页面隐藏时：

```text
停止轮询
```

页面重新可见时：

```text
立即补查一次
```

### 13.7 deadline 逻辑接入

deadline 未完成项应改为基于状态判断：

| 状态 | 是否提醒 |
| --- | --- |
| pending_decision | 是 |
| selected | 是 |
| draft_created | 是 |
| review_pending | 是 |
| documented | 否 |
| ignored | 否 |
| archived | 否 |

其中：

- `selected` / `draft_created` / `review_pending` 都应被视为“跟进未闭环”
- `review_pending` 应在 deadline 中单独提示为：

```text
已检测到文档更新，但尚未确认提交
```

### 13.8 推送事件接入

Phase 13.1 中飞书推送只接入这些事件：

1. `task_selected`
2. `document_confirmed`
3. `soft_deadline`
4. `hard_deadline`
5. `yesterday_carryover`

`draft_detected_updated` 默认不即时推送飞书，只做页面提醒。

### 13.9 验收标准

当以下条件全部满足，Phase 13.1 可以算完成：

1. 页面可创建草稿，并记录 `draft_created_at` 和 `draft_initial_hash`
2. 外部工具修改 Markdown 后，系统能检测到并切换到 `review_pending`
3. 页面能明确提示“待确认知识库文档”
4. 点击“确认提交文档”后，状态变为 `documented`
5. deadline 只提醒未闭环任务
6. `ignored` 不进入 deadline 未完成列表
7. 关键操作事件能写入 `task_event`

### 13.10 开发顺序

推荐顺序：

1. 数据库 schema 变更
2. 后端检测 API
3. 前端状态和提示
4. 手动刷新检测
5. 90 秒轮询
6. deadline 逻辑接入
7. 飞书推送接入
